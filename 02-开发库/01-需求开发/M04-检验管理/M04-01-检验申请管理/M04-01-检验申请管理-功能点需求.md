## 1. 功能概述

### 1.1 功能定位

检验申请管理是检验管理子系统(LIS)的入口模块，负责接收门诊/住院医生开立的检验申请、管理申请状态流转、支持申请查询追踪、申请单打印、申请撤回处理等核心功能。系统实现检验申请的全流程管理，为后续标本采集、检验执行、报告发布奠定基础。

### 1.2 业务目标

| 目标类型 | 目标描述 | 衡量指标 |
|----------|----------|----------|
| 效率目标 | 实现检验申请无缝接收 | 申请接收响应≤2秒 |
| 追踪目标 | 申请状态全程追踪 | 申请追踪覆盖率100% |
| 安全目标 | 申请撤回权限控制 | 撤回操作100%审计记录 |
| 集成目标 | 与门诊/住院系统集成 | 申请接收成功率≥99% |

## 3. 用户故事

### US-M04-01-001 检验申请接收

**故事卡片**
```
作为 检验科工作人员
我想要 接收门诊/住院的检验申请
以便于 开始处理患者的检验请求
```

**优先级**: P0（MVP必需）

**故事点**: 5

**验收标准 (Gherkin)**
```gherkin
Feature: 检验申请接收

  Scenario: 接收门诊检验申请
    Given 门诊医生已开立检验申请"血常规"
    And 患者已完成缴费
    When 检验系统接收申请
    Then 系统创建检验申请记录，状态为"待采集"
    And 申请记录包含患者信息、检验项目、申请科室
    And 患者信息包含姓名、性别、年龄、患者编号

  Scenario: 接收住院检验申请
    Given 住院医生已开立检验申请"肝功能"
    And 护士站已确认执行
    When 检验系统接收申请
    Then 系统创建检验申请记录，状态为"待采集"
    And 申请记录关联住院号和床号

  Scenario: 申请状态追踪
    Given 检验申请已创建
    When 工作人员查询申请状态
    Then 系统显示申请当前状态
    And 显示状态变更历史时间线
    And 显示各环节处理人信息

  Scenario: 申请信息查询
    Given 检验科有待处理的申请
    When 工作人员输入查询条件
    Then 系统显示符合条件的申请列表
    And 支持按申请编号、患者姓名、状态筛选
    And 支持按申请日期范围筛选

  Scenario Exception: 申请信息不完整
    Given 接收到的申请缺少必要信息
    When 系统校验申请数据
    Then 系统标记为"待补全"
    And 系统通知申请科室补充信息

  Scenario Boundary: 申请有效期校验
    Given 检验申请创建已超过24小时
    And 申请状态仍为"待采集"
    When 系统执行定时检查
    Then 系统发送超时提醒
    And 标记为"待处理超时"
```

### US-M04-01-002 申请撤回处理

**故事卡片**
```
作为 临床医生
我想要 撤回未采集的检验申请
以便于 调整检验方案或取消不必要的检验
```

**优先级**: P1（重要）

**故事点**: 3

**验收标准 (Gherkin)**
```gherkin
Feature: 申请撤回处理

  Scenario: 正常撤回流程
    Given 检验申请状态为"待采集"
    And 未进行标本采集
    And 用户有撤回权限
    When 用户选择撤回申请
    And 填写撤回原因
    Then 申请状态变更为"已撤回"
    And 系统记录撤回人、撤回时间、撤回原因
    And 系统通知申请科室

  Scenario: 已缴费申请撤回
    Given 患者已缴费的申请被撤回
    When 撤回完成
    Then 系统触发退款流程
    And 通知财务系统处理退款

  Scenario Exception: 已采集不可撤回
    Given 检验申请状态为"已采集"

    Then 系统提示"标本已采集，不可撤回"

    When 用户尝试撤回申请
    Then 系统提示"无撤回权限"
```

### 7.2 关键业务规则详解

#### BR-LIS-013: 检验申请状态流转

检验申请状态流转规则：待缴费(1) → 待采集(2) → 已采集(3) → 运送中(4) → 已接收(5) → 检验中(6) → 已完成(7)

**状态流转约束**:
- 待缴费状态可流转为待采集、已撤回
- 待采集状态可流转为已采集、已撤回
- 已采集及后续状态不可撤回
- 已完成、已撤回为终态，不可变更

**触发时机**: 状态变更时

**校验逻辑**:
```
valid_transitions = {
    1: [2, 8],     // 待缴费 -> 待采集/已撤回
    2: [3, 8],     // 待采集 -> 已采集/已撤回
    3: [4],        // 已采集 -> 运送中
    4: [5],        // 运送中 -> 已接收
    5: [6],        // 已接收 -> 检验中
    6: [7],        // 检验中 -> 已完成
    7: [],         // 已完成 -> 终态
    8: []          // 已撤回 -> 终态
}

IF target_status NOT IN valid_transitions[current_status] THEN
    RETURN error("状态流转不合法")
END IF

// 记录状态变更
INSERT INTO lis_request_status_log (request_id, from_status, to_status, operator, operate_time)
VALUES (:request_id, :current_status, :target_status, :current_user, NOW())
```

#### BR-LIS-031: 门诊检验申请接收

门诊检验申请接收规则：接收门诊系统推送的检验申请，创建检验申请记录

**触发时机**: 接收门诊申请消息时

**集成逻辑**:
```
// 接收消息
message = RECEIVE_message('OUTPATIENT_LAB_REQUEST')

// 解析消息
request_data = PARSE_json(message.body)

// 校验必填字段
required_fields = ['patient_id', 'patient_name', 'dept_id', 'doctor_id', 'items[]']
FOR each field IN required_fields DO
    IF request_data[field] IS NULL THEN
        RETURN error("缺少必填字段：" + field)
    END IF
END FOR

// 创建检验申请
request_id = INSERT INTO lis_request (
    source_type,      -- 1=门诊
    encounter_id,     -- 门诊就诊ID
    patient_id,
    patient_name,
    ...
    request_status,   -- 初始状态：待缴费
    is_urgent
) VALUES (1, ...)

// 发送确认消息
SEND_message('LIS_REQUEST_RECEIVED', {'request_id': request_id, 'request_no': generate_request_no()})
```

#### BR-LIS-009: 标本采集超时校验

标本采集超时校验规则：检验申请创建后超过规定时间未采集，发送超时提醒

**触发时机**: 定时检查（每30分钟）

**超时时间**: 4小时

**校验逻辑**:
```
timeout_hours = 4

overdue_requests = SELECT * FROM lis_request
                   WHERE request_status = 2  -- 待采集
                     AND TIMESTAMPDIFF(HOUR, create_time, NOW()) >= timeout_hours

FOR each request IN overdue_requests DO
    // 发送超时提醒
    SEND_notification({
        'to': request.dept_id,
        'type': 'COLLECTION_TIMEOUT',
        'message': '检验申请已超过4小时未采集'
    })
    
    // 标记为超时
    UPDATE lis_request SET is_timeout = 1, timeout_time = NOW() WHERE request_id = request.request_id
END FOR
```

## 9. 附录

### A. 申请编号生成规则

```
申请编号格式: LJ + YYYYMMDD + 序号(4位)

示例: LJ202606190001

生成逻辑:
1. 获取当前日期字符串：YYYYMMDD
2. 查询当日最大序号
3. 最大序号+1，格式化为4位
4. 拼接生成完整编号

伪代码:
date_str = DATE_FORMAT(NOW(), '%Y%m%d')
max_seq = SELECT MAX(CAST(SUBSTRING(request_no, 11, 4) AS UNSIGNED))
          FROM lis_request
          WHERE request_no LIKE CONCAT('LJ', :date_str, '%')
next_seq = max_seq + 1 IF max_seq IS NOT NULL ELSE 1
request_no = 'LJ' + date_str + LPAD(next_seq, 4, '0')
```
