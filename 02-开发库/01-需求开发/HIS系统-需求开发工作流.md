- **ID**: his-requirement-development-workflow

## 工作流概述

本工作流用于持续开发HIS系统各模块的需求文档，采用产品经理agent模型的标准方法论：
- **需求分析方法**: INVEST原则、MoSCoW优先级、RICE评分
- **文档标准**: PRD模板、用户故事模板、验收标准Gherkin格式
- **质量标准**: 需求覆盖率≥95%、INVEST符合度100%

## 工作流阶段定义

### Phase 1: 需求理解与分析
**目标**: 深入理解模块PRD，提取功能点，分析业务场景

**输入**:
- 模块PRD文档
- 全局需求概览
- 模块划分文档
- 数据字典

**输出**:
- 功能点清单
- 业务场景分析
- 用户角色识别
- 依赖关系梳理

**检查清单**:
- [ ] 所有功能点已识别
- [ ] 业务场景覆盖完整
- [ ] 用户角色定义清晰
- [ ] 上下游依赖明确

### Phase 2: 用户故事编写
**目标**: 将功能点转化为符合INVEST原则的用户故事

**输入**:
- 功能点清单
- 业务场景分析
- 用户角色定义

**输出**:
- 用户故事集(US-XXX-XXX格式)
- 故事优先级排序(RICE评分)
- 故事依赖关系图

**INVEST检查**:
- Independent: 故事之间尽量独立
- Negotiable: 故事可协商调整
- Valuable: 对用户有价值
- Estimable: 可估算工作量
- Small: 足够小可在一个Sprint完成
- Testable: 可测试验证

### Phase 3: 验收标准定义
**目标**: 为每个用户故事编写Gherkin格式验收标准

**输入**:
- 用户故事集
- 业务规则文档

**输出**:
- 验收标准文档(AC-XXX格式)
- 测试场景覆盖
- 边界条件定义

**Gherkin格式**:
```gherkin
场景: {场景名称}
  Given {前置条件}
  When {用户操作}
  Then {预期结果}
  And {附加条件}
```

### Phase 4: 业务规则梳理
**目标**: 梳理模块业务规则，形成结构化业务规则文档

**输入**:
- PRD业务规则
- 用户故事
- 验收标准

**输出**:
- 业务规则文档(BR-XXX格式)
- 规则触发条件
- 规则执行逻辑
- 异常处理规则

**规则分类**:
- 数据校验规则
- 业务流转规则
- 权限控制规则
- 计算规则
- 集成规则

### Phase 5: 质量检查与完善
**目标**: 检查需求文档质量，确保完整性和一致性

**检查项**:
- [ ] 需求覆盖率 ≥ 95%
- [ ] INVEST符合度 = 100%
- [ ] 验收标准可执行
- [ ] 业务规则无冲突
- [ ] 文档格式规范

## 产出物规范

### 目录结构
```
02-开发库/01-需求开发/{模块编号}-{模块名称}/
├── {模块编号}-{模块名称}-PRD.md          # 产品需求文档
├── {模块编号}-{模块名称}-用户故事.md      # 用户故事集
├── {模块编号}-{模块名称}-验收标准.md      # 验收标准文档
├── {模块编号}-{模块名称}-业务规则.md      # 业务规则文档
└── {子模块编号}-{子模块名称}/
    └── {子模块编号}-{子模块名称}-功能点需求.md  # 功能点详细需求
```

### 文档命名规范
- PRD: `{模块编号}-{模块名称}-PRD.md`
- 用户故事: `{模块编号}-{模块名称}-用户故事.md`
- 验收标准: `{模块编号}-{模块名称}-验收标准.md`
- 业务规则: `{模块编号}-{模块名称}-业务规则.md`
- 功能点需求: `{子模块编号}-{子模块名称}-功能点需求.md`

### ID命名规范
- 用户故事: `US-{模块编号}-{序号}` (例: US-M09-001)
- 验收标准: `AC-{用户故事ID}-{场景序号}` (例: AC-US-M09-001-01)
- 业务规则: `BR-{模块编号}-{分类}-{序号}` (例: BR-M09-VAL-001)
- 功能点: `F-{模块编号}-{子模块序号}-{功能序号}` (例: F-M09-01-01)

### 需求文档质量指标
| 指标 | 目标值 | 检查方法 |
|------|--------|----------|

### 文档格式规范

- 表格对齐规范
- 标题层级清晰(最多4级)
- 代码块标注语言
- 链接使用相对路径

## 协作关系

### 输入依赖
- 项目管理: 项目目标、时间计划
- 架构设计: 技术架构、模块划分
- 业务方: 业务需求、业务规则

- 研发工程师: PRD、用户故事、验收标准
- 测试工程师: 验收标准、测试用例输入

## 注意事项

1. **需求变更管理**: 需求变更需评估影响范围，记录版本历史
2. **依赖模块协调**: 关注上下游模块依赖，确保接口一致
3. **业务规则冲突**: 发现冲突及时与业务方确认
4. **文档版本控制**: 每次更新记录变更历史
5. **质量检查**: 每个阶段完成后进行质量检查
