# ADR-M09-001: 数据权限控制方案

## 决策编号

ADR-M09-001

## 状态

已接受

## 上下文

数据权限控制是医疗信息安全的核心。根据PRD和业务规则文档：

**业务要求**：
- 权限分为菜单权限、按钮权限、数据权限三级（BR-SYS-004）
- 病历查阅按角色分级控制（BR-EMR-004）
- 敏感数据操作必须记录操作前后值（BR-SYS-009）
- 满足等保三级安全要求

**权限场景**：
1. 医生只能查看本科室患者病历
2. 护士只能查看本病区患者信息
3. 管理者只能查看脱敏统计数据
4. 会诊医生需要临时授权

## 决策

采用**RBAC三级权限+数据范围控制**的权限管理方案：

### 架构设计

```
┌─────────────────────────────────────────────────────────────────────┐
│                    数据权限控制架构                                   │
│                                                                     │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │                    权限模型                                   │  │
│  │                                                              │  │
│  │    用户 ─→ 角色 ─→ 权限                                      │  │
│  │                      │                                       │  │
│  │                      ▼                                       │  │
│  │         ┌────────────────────────┐                          │  │
│  │         │    权限三级结构         │                          │  │
│  │         │  ┌──────────────────┐  │                          │  │
│  │         │  │ 1. 菜单权限       │  │ 页面可见性               │  │
│  │         │  │ 2. 按钮权限       │  │ 操作可用性               │  │
│  │         │  │ 3. 数据权限       │  │ 数据范围                 │  │
│  │         │  └──────────────────┘  │                          │  │
│  │         └────────────────────────┘                          │  │
│  └──────────────────────────────────────────────────────────────┘  │
│       │                                                             │
│       ▼                                                             │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │                    数据权限范围                               │  │
│  │  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │  │
│  │  │ 全部数据    │ │ 本科室数据  │ │ 本病区数据  │            │  │
│  │  │ ALL         │ │ DEPT        │ │ WARD        │            │  │
│  │  └─────────────┘ └─────────────┘ └─────────────┘            │  │
│  │  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │  │
│  │  │ 本人数据    │ │ 自定义数据  │ │ 仅授权数据  │            │  │
│  │  │ SELF        │ │ CUSTOM      │ │ AUTHORIZED  │            │  │
│  │  └─────────────┘ └─────────────┘ └─────────────┘            │  │
│  └──────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

### 权限模型设计

```java
/**
 * 数据权限范围枚举
 */
public enum DataScopeEnum {
    
    ALL(1, "全部数据", "查看所有数据"),
    DEPT(2, "本科室数据", "仅查看本科室数据"),
    DEPT_AND_SUB(3, "本科室及下级", "查看本科室及下级科室数据"),
    WARD(4, "本病区数据", "仅查看本病区数据"),
    SELF(5, "本人数据", "仅查看本人创建/负责的数据"),
    CUSTOM(6, "自定义", "自定义数据范围"),
    AUTHORIZED(7, "仅授权数据", "仅查看被授权的数据");
    
    private final Integer code;
    private final String name;
    private final String description;
}
```

### MyBatis数据权限拦截器

核心实现：通过MyBatis拦截器自动注入数据权限过滤SQL，对业务层透明。

## 后果

### 正面影响
1. RBAC标准模型成熟稳定
2. 三级权限灵活控制
3. 数据范围控制自动注入SQL

### 负面影响
1. 数据权限注入增加查询复杂度
2. 角色配置工作量大

### 应对措施
- 优化数据库索引
- 提供默认角色模板

## 合规性

| 标准/规范 | 符合程度 | 说明 |
|-----------|----------|------|
| BR-SYS-004 | 完全符合 | 三级权限控制 |
| 等保三级 | 符合 | 数据权限隔离 |

# ADR-M09-002: 审计日志方案

## 决策编号

ADR-M09-002

## 状态

已接受

## 上下文

YUDAO-AI-HIS需要完善的审计日志机制。根据业务规则：

**业务要求**：
- 敏感数据操作必须记录操作前后值（BR-SYS-009）
- 登录日志保留≥6个月
- 操作日志保留≥3年
- 满足等保三级审计要求

## 决策

采用**集中式审计日志**：

### 日志类型与保留期限

| 日志类型 | 保留期限 | 说明 |
|----------|----------|------|
| 登录日志 | ≥ 6个月 | 用户登录/登出记录 |
| 操作日志 | ≥ 3年 | 业务操作记录 |
| 敏感数据访问日志 | ≥ 3年 | 敏感数据查看记录 |
| 异常日志 | ≥ 1年 | 系统异常记录 |

## 后果

### 正面影响
1. 满足等保三级审计要求
2. 操作可追溯，支持安全分析

### 负面影响
1. 日志量大，需要分表存储
2. 日志写入影响性能

### 应对措施
- 日志表按月分表
- 异步写入日志

## 合规性

| 标准/规范 | 符合程度 | 说明 |
|-----------|----------|------|
| BR-SYS-009 | 完全符合 | 记录操作前后值 |
| 等保三级 | 完全符合 | 审计日志要求 |

# ADR-M09-003: 电子病历签名与归档

## 决策编号

ADR-M09-003

## 状态

已接受

## 上下文

电子病历签名与归档是医疗法律效力和病历管理的关键功能。根据PRD和业务规则文档：

**业务要求**：
- 电子签名必须符合《电子签名法》，支持CA数字签名（BR-EMR-002）
- 病历归档后不可修改（BR-EMR-001）
- 医疗纠纷时病历封存需双方确认（BR-EMR-003）
- 病历保存期限：门诊≥15年，住院≥30年

## 决策

采用**CA数字签名+归档锁定**的电子病历管理架构：

### 病历状态机

```
草稿 ─→ 已签名 ─→ 已审核 ─→ 已归档 ─→ 已封存
```

### 核心功能
1. CA数字签名：符合《电子签名法》要求
2. 时间戳服务：第三方时间戳提供时间证明
3. 归档锁定：归档后禁止修改
4. 封存机制：医疗纠纷场景下双方确认封存

## 后果

### 正面影响
1. CA数字签名具有法律效力
2. 归档锁定防止篡改
3. 封存机制支持医疗纠纷场景

### 负面影响
1. 第三方CA服务需要费用
2. 归档PDF占用存储空间

### 应对措施
- 预算采购CA服务
- 使用对象存储归档文件

## 合规性

| 标准/规范 | 符合程度 | 说明 |
|-----------|----------|------|
| 《电子签名法》 | 完全符合 | CA数字签名 |
| BR-EMR-001~003 | 完全符合 | 归档、签名、封存 |
| 病历保存期限 | 符合 | 门诊≥15年，住院≥30年 |
