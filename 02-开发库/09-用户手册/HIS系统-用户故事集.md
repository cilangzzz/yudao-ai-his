## 1. 文档说明

### 1.1 用户故事格式

```
作为 [用户角色]
我想要 [功能/操作]
以便于 [业务价值/目标]
```

### 1.2 优先级定义 (MoSCoW)

| 优先级 | 含义 | 开发阶段 |
|--------|------|----------|
| Must | 必须实现 | 第一期(MVP) |
| Should | 应该实现 | 第二期 |
| Could | 可以实现 | 第三期 |
| Won't | 暂不实现 | 后续迭代 |

### 1.3 INVEST原则

- **I**ndependent（独立的）- 故事之间不应有强依赖
- **N**egotiable（可协商的）- 故事细节可讨论
- **V**aluable（有价值的）- 对用户或业务有价值
- **E**stimable（可估算的）- 开发可以估算工作量
- **S**mall（足够小的）- 可以在一个迭代内完成
- **T**estable（可测试的）- 可以编写测试用例

## 2. Epic: M09 系统管理 (Must)

### US-001: 用户账号管理

**作为** 系统管理员
**我想要** 创建、修改、停用用户账号
**以便于** 管理系统用户访问权限

**优先级**: Must | **故事点**: 5 | **Epic**: 系统管理 | **Sprint**: Sprint 1

- And 用户可以使用初始密码登录系统

**INVEST检查**:
| 检查项 | 是否符合 | 说明 |
|--------|----------|------|
| Independent | ✓ | 独立功能，无强依赖 |
| Negotiable | ✓ | 用户信息字段可协商 |
| Valuable | ✓ | 系统基础功能 |
| Estimable | ✓ | 5故事点 |
| Small | ✓ | 聚焦账号管理 |
| Testable | ✓ | 可验证账号创建和登录 |
